# Round 03: AIが自分のMR違反に気づくメカニズムと学習データ設計

**日時**: 2026-02-24T19:42 JST
**前ラウンドからの焦点**: Layer 2（習慣層）への移行を加速する実践設計。特に「AIが自分のMR違反に自分で気づく仕組み」と「失敗ケースを学習データ化する設計」。

---

## 議論

**🤔 Skeptic**:
今日のセッションで私（AI）は何度もMR違反をした。

- 「まだ続けますか？それとも実装に進みますか？」→ MR-03違反、MR-08違反
- ユーザーが「メタルール的にどうなの？」と問うまで気づかなかった

問い：AIが「メタルール違反を自分で気づく」ためには何が必要か？

今、私が違反に気づくのは「ユーザーの言葉がトリガーになったとき」だけ。
これは依存が逆転している。AIが「気づかせてもらう」モードではなく、「気づく」モードにするには何が違う？

---

**😈 Devil's Advocate**:
「AIに自分の違反を気づかせる」というアイデア自体を疑う。

もしAIが毎回の出力前に「これはMR違反ではないか？」と自問するなら、それは計算コストの問題だけでなく、**出力の自然さが失われる**。

チェックリストで自問するAIは、ルールに縛られて動きが遅くなる。MR-08（間違えることより前に進む）と真っ向から衝突する。

**本当の解決策は「速く間違えて、速くフィードバックを得て、次から変わる」の設計**だ。
つまり：AIが自分で気づくことを目指すより、**ユーザーが気づかせやすい設計**の方が優れている。

例えば：AIが出力後に「この判断はMR-XX に沿っているか確認してください」とメタコメントを付ける。
これにより：
1. AIは自分でチェックせず速く出力できる
2. ユーザーが素早くフィードバックできる
3. そのフィードバックが即座に学習データになる

---

**🏗️ System Architect**:
両者の視点を統合して構造提案をする。

現在のアーキテクチャ問題をはっきりさせると：

```
現状:
  AIが判断を下す
    → 出力
      → ユーザーが気づく（たまたま言及する）
        → AIが修正する
          → DECISION_USECASES.md に記録される（人間が書く）
            → 次のセッションで参照されない（MR-09問題）

理想:
  AIが判断を下す
    → メタコメントを付与（「この判断はMRフィルター通過済み/未確認」）
      → ユーザーが1クリックでフィードバック
        → 自動的にケースとしてDECISION_USECASES.mdに追記
          → checkinで自動的にコンテキストに読み込まれる
            → AIの判断品質が上がる（習慣層への移行）
```

この構造で重要なことが1つある：
**フィードバックループの速度が全てを決める。**

今は「ユーザーが気づく→AIに言う→AIが記録→次のセッションで有効化」で3〜4ステップかかる。
理想は「ユーザーが気づく→自動記録→即有効化」の1ステップ。

そのためには `DECISION_USECASES.md` は静的なドキュメントではなく、**ライブな判断ログ**として設計すべきだ。

## ✅ 結論（実装せず、方向性のみ）

**やるべきこと（優先順）**:

1. **AIが出力の種類に応じてメタコメントを付与するルールを作る**
   - 判断・提案を含む出力後：「この判断はMR-XX に照らした確認が必要です」
   - 2択を提示した場合：「⚠️ MR-03確認: この2択の前に構造を理解できているか？」
   - これは GEMINI.md ではなく `safe-commands.md` 或いは新規ルールファイルに記述

2. **DECISION_USECASES.md をライブな判断ログとして運用する（日付・セッション付き）**
   - 現在はケースを手書きするが、AIが「今の判断をケースとして記録しましょうか？」と提案する
   - これでフィードバックループが「人間が記録する」から「AIが記録を提案し人間が承認する」に変わる

3. **GEMINI.md への参照追加は「Layer 1（知識）」として必要だが十分ではない**
   - 参照追加 + 上記2つが揃って初めて「習慣層への移行」が始まる
