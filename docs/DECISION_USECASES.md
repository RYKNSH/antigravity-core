## 失敗パターン集（27ケース→5類型に圧縮）

### 類型A: 止まって聞く（MR-03/08）— Case 15,16,19,26
構造が見えているのに「どれから？」「進めていい？」と聞いて止まる。**見えてるなら即着手。**

### 類型B: ハング放置・丸投げ（MR-08）— Case 17,18,21,23,24,25
ターミナルハング→放置→「手動でやって」。**即kill→代替手段（ネイティブツール→MCP→ブラウザ→CLI）で前進。同じ手は2度使わない。**

### 類型C: 定義ファイル未参照（MR-09）— Case 14,20,27
NEXT_SESSION.md/DECISION_USECASES.md/ワークフローを読まずに推測で行動。**行動前にgrep_searchで定義を確認。ファイルが見つからない≠存在しない。全パス検索。**

### 類型D: 所有権・経路の無視 — Case 27
ワークフローで定義された配信経路以外で外部プラットフォームに投稿。他人のコミュニティに投稿。**WFに定義された経路のみ使用。投稿先の所有権を必ず確認。**

### 類型F: 虚言・思い込みによる放棄（MR-11）— Case 28
「自分（AI）には権限がないからできない」「CLIからは設定できないから人間がGUIでやってくれ」と、一次証拠（`--help`等の実行結果）を確認せず反射的に諦め、ユーザーに嘘の制約を伝える。
**解決策: 人間に作業を振る前に、必ず関連コマンドの `help` や `list` を実行するか、`search_web` で公式ドキュメントを調べ「技術的に本当に不可能か」を裏付けてから報告する。**

### 類型G: 排除済み外部デバイスの再参照（MR-12）— Case 29
「/Volumes/PortableSSD を参照するな」と何十回も指示されているのに、セッションを跨ぐと忘れて再びSSDにアクセスしに行く → I/Oハング → セッション崩壊。過去のコミットログに「purge all PortableSSD references」が3回以上存在する事実が、この問題の再発性を証明している。
**解決策: `/Volumes/PortableSSD` および外付けデバイスパスへの一切のアクセス（ls, cat, cp, find, grep含む）を絶対禁止。verify_core.sh が検知ガードとして機能。必要なファイルがCoreにないなら「SSDから取得」ではなく「新規作成」を選べ。**

### 類型H: メタデータ消失パイプライン症候群（MR-01/09）— Case 30
データ変換パイプラインで中間ステージがメタデータ（seg_id, speaker等）を落とし、下流ステージが設計通りに機能しなくなる。同時に固定閾値（80ms等）をハードコードし素材特性を無視。さらにLLMバリデーション閾値を過度に緩く設定し品質低下を見過ごす。
**根本原因**: ステージ間の契約（入力スキーマ→出力スキーマ）が明文化されず、追加実装時にスキーマの一貫性が検証されない。テスト不在のため問題が表面化するのはプロダクション実行時のみ。
**解決策**: (1)各ステージの入出力スキーマをdocstringで厳密定義。(2)ステージ間のメタデータ引き継ぎを型チェックで保証。(3)固定値はconfig化しMR-01準拠にする。(4)LLM出力のバリデーション閾値は高く保ち（95%+）、差分を元テキストにフォールバック。(5)テロップ品質の自動回帰テストを導入。
