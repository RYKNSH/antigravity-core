# 持ち運べる、の幻想。


「ポータブル」って言葉、
好きだったんです。

どこにでも持っていける。
差し込めばすぐ動く。
自分の環境が、いつでもそこにある。

でもそれ、本当に「持ち運べてた」んでしょうか。



ぼくは、開発環境をSSDに入れてました。

外付けのSSD。
ワークフロー、スクリプト、設定ファイル、ブログ記事。
ぜんぶ1つのドライブに詰め込んで。

職場でも、自宅でも、
SSDを差し込めば同じ環境が立ち上がる。

そう信じてた。



ところが、ある日気づいたんです。

SSDの読み込みが遅くなってきた。
ファイル検索に2分以上かかる。
コマンドがタイムアウトする。

おかしいな、と思いました。

でも「SSD = 速い」という先入観があるから、
原因をべつの場所に探してしまう。

スクリプトのタイムアウト値を伸ばしたり、
メモリ管理を強化したり。

SSDそのものが問題だとは、なかなか思えなかった。

これが盲点、というやつですね。



もうひとつ、もっと怖いことが起きました。

開発環境をGitHubに移行しよう、と決めたとき。
コードをオープンソースとして公開しようとしたんです。

その作業の途中で、気づいた。

設定ファイルのなかに、
GitHub APIの認証トークンが入っていた。

しかも、すでにコミット履歴に残っている。

つまり、公開した瞬間に、
ぼくのGitHubアカウントへのフルアクセスが
世界中に晒されるところだった。

シークレット、というのは
システムの裏口の鍵みたいなものです。
これが漏れると、他人がぼくのふりをしてコードを書き換えたり、
非公開のプロジェクトを覗いたりできてしまう。

SSDに入れて持ち歩いている間は、
「物理的に手元にある」から安全だと思っていた。

でもネットワークに出した瞬間、
その安心感は幻想だったと分かった。



ここで学んだことは、2つあります。

ひとつは、「持ち運ぶ」と「同期する」はちがうということ。

SSDを物理的に差し込む。
これは「持ち運ぶ」です。

でもGitHubに置いて、どのマシンからでも引っ張ってこれるようにする。
これは「同期する」です。

持ち運ぶ、は物理的な制約に縛られる。
SSDが壊れたら終わり。読み込みが遅くなったら、環境ごと遅くなる。

同期する、はそこから自由になれる。
マシンが変わっても、ネットさえあれば同じ環境が再現される。

ポータブルの本当の意味は、
「モノを持ち歩くこと」じゃなくて、
「状態を再現できること」だったんです。



もうひとつは、
「安全だと感じること」と「安全であること」はちがう、ということ。

SSDが手元にある。物理的に誰も触れない。
だから安全。

そう思っていた。

でも中身に秘密が混ざっていて、
いつかネットワークに出す可能性がある時点で、
それは時限爆弾と同じでした。

結局、コミット履歴をすべて書き換えて、
認証トークンを無効化して、
新しいものに差し替えた。

さらに、秘密の情報と公開してもいい情報を、
物理的にべつのリポジトリ（保管庫）に分離した。

「安全だと感じる」ではなく、
「構造的に漏れない」ようにした。



ぼくの環境は今、こうなってます。

公開リポジトリには、ワークフローとスクリプトだけが入っている。
非公開リポジトリには、トークンやブログ原稿が入っている。
新しいマシンで1つのコマンドを実行すれば、両方が自動で接続されて、
すべてが元どおりに動き始める。

SSDはもう要らない。

壊れても、なくしても、
同じ環境が5分で再現される。



もし今、
「データは手元にあるから大丈夫」
と思っているなら。

ちょっとだけ、立ち止まってみてほしいんです。

それは本当に安全ですか。
それは本当にポータブルですか。

モノを持ち歩くことと、
状態を再現できることは、
ぜんぜんちがう話なんです。



持ち運べる、の本当の意味は、
「手放せる」ということなのかもしれません。
