# 理想系定義（Ideal State Definition）

> **制約を一旦忘れ、真に望む状態を描く。**
> 理想系は北極星。すべての決定は「理想に近づくか」で判断する。

---

## 🌟 プロジェクトの理想系

### 開発体験の理想系

```
理想状態:
- 開発者は技術的な障壁なく、ビジネスロジックに集中できる
- AIエージェントがルーティンワークを代行する
- コードベースは常に一貫性があり、誰が読んでも理解できる
- テストは信頼でき、CIは常にグリーン
- デプロイは1クリックで完了する
```

### ユーザー体験の理想系

```
理想状態:
- ページは瞬時に読み込まれる
- エラーは発生せず、発生しても分かりやすく回復できる
- UI は直感的で、説明なしに操作できる
- アクセシビリティは完全で、誰でも使える
```

### 運用の理想系

```
理想状態:
- システムは24/7稼働し、自己修復する
- 問題はユーザーが気づく前に検知・解決される
- コストは最適化され、無駄がない
- スケールは需要に応じて自動調整される
```

## 🎯 理想系を使った判断

### 判断フレームワーク

変更を加える際、以下を自問する：

| 質問 | 期待される回答 |
|------|--------------|
| この変更は理想系に近づくか？ | Yes |
| この変更は理想系から遠ざけるか？ | No |
| 理想系に近づく別の方法はあるか？ | 検討済み |
| この変更なしで済む方法はあるか？ | 検討済み |

### 判断フローチャート

```
変更を検討
    ↓
理想系に近づく？ ──No──→ 本当に必要？ ──No──→ 却下
    │                        │
   Yes                      Yes
    ↓                        ↓
より良い方法は？ ──Yes──→ 再検討
    │
   No
    ↓
実装へ
```

## 📚 理想系の記録

### [例] 認証システムの理想系

#### 現状の問題
- ログインに3つのステップが必要
- パスワードリセットが複雑
- ソーシャルログインが限定的

#### 理想状態
- ワンクリックで認証完了
- パスワードレス認証（WebAuthn, Passkeys）
- あらゆるIdPとの統合

#### 段階的アプローチ
1. **Phase 1**: 既存フローの最適化（〜2週間）
2. **Phase 2**: ソーシャルログイン追加（〜4週間）
3. **Phase 3**: Passkeys導入（〜8週間）

---

> [!TIP]
> 理想系は固定ではありません。
> 技術の進化や要件の変化に応じて、定期的に見直してください。
